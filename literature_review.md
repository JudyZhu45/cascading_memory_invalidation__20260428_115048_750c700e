# Literature Review: Cascading Memory Invalidation in User Memory Graphs

**Research Hypothesis**: A memory graph with typed dependency edges can propagate invalidation signals to dependent memories without explicit user retraction, via (1) structural LOCATED_IN-type edges for explicit context shifts, and (2) semantic CONFLICTS_WITH edges (inferred via LLM or behavioral co-occurrence) for implicit preference drift.

---

## Research Area Overview

This research sits at the intersection of three fields:
1. **Long-term conversational memory** for LLM agents (retrieval, storage, updating)
2. **User preference evolution / personalization** (tracking changing preferences over time)
3. **Knowledge editing with ripple effects** (propagating updates through structured knowledge graphs)

The core challenge identified across all relevant literature is **belief-update failure**: systems retrieve historically stated preferences correctly but fail to update them when life events or new information should cause them to change. This is fundamentally a graph propagation problem — when a root node changes, downstream dependent nodes must be invalidated and updated.

## Key Papers

### Paper 1: HorizonBench — Long-Horizon Personalization with Evolving Preferences
- **Authors**: Shuyue Stella Li, Bhargavi Paranjape, Kerem Oktar, et al.
- **Year**: 2026 (April 19, preprint under review)
- **Source**: arXiv:2604.17283
- **GitHub**: https://github.com/stellalisy/HorizonBench
- **Dataset**: stellalisy/HorizonBench (HuggingFace)
- **Key Contribution**: Introduces the **mental state graph** with typed dependency edges for preference evolution tracking. Defines "long-horizon personalization" as the task of tracking preferences that evolve via life events over 6-month timelines. Identifies **belief-update failure** as the primary bottleneck.
- **Methodology**:
  - Mental state graph G: nodes = preferences, social connections, life events; edges = typed dependency edges (e.g., dietary preferences → restaurant choices → social outing plans)
  - Preference evolution: life events propagate changes through dependency edges to downstream nodes; with p_evo=0.15, a single event alters 2–5 preferences simultaneously (cascading updates)
  - Expression tracking: separate timestamps for when preferences evolve vs. when they are expressed in conversation
  - Benchmark: 4,245 5-option multiple-choice items from 360 simulated users, 6-month histories averaging ~163K tokens
  - Hard-negative distractors: pre-evolution preference values injected as distractors
  - 5-LLM consensus filter: items pass only if 5 validators all fail without conversation history
- **Datasets Used**: Synthetic (generated by Claude-sonnet-4.5, o3, Gemini-3-flash)
- **Results**: Best model (Claude-opus-4.5) 52.8%; most models at or below 20% chance baseline. All 25 models select pre-evolution distractor >25% of wrong answers (belief-update failure). Gap between evolved and static preferences: −2.7pp avg across 25 models.
- **Controlled Experiments**:
  - Context length: belief-update failure persists even with 0–14 day history (short horizon), intensifies (−5.7pp gap)
  - Expression explicitness: implicit expression worsens performance, distractor rate rises from 49.7% → 58.8%
  - Counterfactual subtlety: gap persists even with easier options (22/25 models negative)
- **Code Key Files**:
  - `src/causal_framework/evolution/relation_propagator.py`: Graph propagation mechanism (cascading updates)
  - `src/causal_framework/evolution/preference_evolve_manager.py`: Orchestrates stability bias, context adaptation, experience-driven change, relational interdependence
  - `src/causal_framework/models/preference_model.py`: Preference node representation with evolution history
  - `src/pipeline/generation_pipeline.py`: Full pipeline for generating benchmark items from mental state graph
- **Relevance**: **Most directly relevant paper** — implements the exact mental state graph with typed dependency edges described in the research hypothesis. The RelationPropagator class implements cascading invalidation with configurable propagation depth (default=2), decay factor (0.7), and strength threshold (0.2). The preference evolution provenance tracking is the ground truth for evaluating invalidation correctness.

### Paper 2: LoCoMo — Evaluating Very Long-Term Conversational Memory of LLM Agents
- **Authors**: Adyasha Maharana, Dong-Ho Lee, Sergey Tulyakov, Mohit Bansal, Francesco Barbieri, Yuwei Fang
- **Year**: 2024 (Feb 27, ACL 2024)
- **Source**: arXiv:2402.17753
- **Dataset**: snap-research/locomo (50 dialogues, 300 turns avg, 9K tokens)
- **Key Contribution**: First large-scale very long-term dialogue dataset (LoCoMo). Uses temporal event graphs to scaffold conversations and ground memory.
- **Methodology**:
  - LLM-based generative agents + human annotation for long-range consistency
  - Each speaker has: unique persona, timeline of causally interlinked events, reflect & respond mechanism
  - Three evaluation tasks: (1) QA (single-hop, multi-hop, temporal, commonsense, adversarial), (2) event graph summarization, (3) multimodal dialog generation
- **Datasets Used**: Self-generated (50 conversations, 35 sessions max, human-verified)
- **Results**: Long-context LLMs improve memory by 22-66% vs. base but lag behind human performance by 56%. RAG offers balanced compromise, especially when dialogues transformed to speaker-observation databases. LLMs struggle with temporal reasoning (73% below human) and adversarial questions (83% below base model).
- **Relevance**: Primary evaluation dataset. PAMU (2510.09720) was evaluated on LoCoMo. LoCoMo's temporal event graphs are the structural analog to the typed dependency edges in the research hypothesis. The QA benchmark covers temporal reasoning needed for cascading invalidation evaluation. **This is the user-specified benchmark dataset.**

### Paper 3: MemGPT — Towards LLMs as Operating Systems
- **Authors**: Charles Packer, Sarah Wooders, Kevin Lin, Vivian Fang, Shishir G. Patil, Ion Stoica, Joseph E. Gonzalez
- **Year**: 2024 (Oct 2023, Feb 2024 v2)
- **Source**: arXiv:2310.08560
- **Code**: https://research.memgpt.ai
- **Key Contribution**: OS-inspired hierarchical memory management for LLMs — virtual context management via paging between main context and external storage.
- **Methodology**:
  - Main context (prompt tokens): system instructions (read-only) + working context (read-write, key facts/preferences) + FIFO queue (conversation history)
  - External context: recall storage (message database) + archival storage (arbitrary text objects)
  - Queue manager: handles context overflow via queue eviction policy, generates recursive summaries
  - Function executor: LLM-controlled memory management via function calls (read/write/retrieve)
  - Working context stores user preferences and persona for conversational agents
- **Relevance**: MemGPT's working context is the target location for cascading invalidated memories. When a LOCATED_IN or CONFLICTS_WITH edge is triggered, the working context update mechanism is what gets called. MemGPT lacks explicit typed dependency edges and cascading invalidation — this is the research gap. MemGPT also evolved into the Letta platform, which provides production-level persistent memory.

### Paper 4: PersonaMem-v2 — Personalized Intelligence via Learning Implicit User Personas and Agentic Memory
- **Authors**: Bowen Jiang, Yuan Yuan, Maohao Shen, et al.
- **Year**: 2025 (Dec 7)
- **Source**: arXiv:2512.06688
- **Dataset**: bowen-upenn/PersonaMem-v2 (HuggingFace)
- **Key Contribution**: State-of-the-art LLM personalization dataset with 1,000 realistic user-chatbot interactions, 20,000+ implicit user preferences, 128K-token contexts. Introduces agentic memory framework achieving 55% accuracy with 2K-token memory (16× fewer tokens than full context).
- **Methodology**:
  - 1,000 richly detailed personas spanning demographic diversity, mental/physical health
  - 300+ conversation topics, mostly implicit preference revelation
  - RFT (Reinforcement Fine-Tuning) with GRPO: trains 4B-parameter model to 53% accuracy
  - Agentic memory: distills conversation history into compact 2K-token human-readable memory; updates persistently across sessions
  - Stereotypical, anti-stereotypical, and neutral preferences; dynamic evolving preferences across sessions
- **Results**: Frontier LLMs: 37-48% accuracy on implicit personalization. RFT Qwen3-4B: 53%. Agentic memory: 55% (best). Reasoning is the bottleneck, not context length.
- **Relevance**: Demonstrates that compact persistent memory (2K tokens) with RFT-based updating outperforms full-context approaches. The agentic memory framework is a natural candidate for incorporating typed dependency edges and cascading invalidation. The preference evolution tracking directly parallels the research hypothesis.

### Paper 5: RippleEdits — Evaluating the Ripple Effects of Knowledge Editing in Language Models
- **Authors**: Roi Cohen, Eden Biran, Ori Yoran, Amir Globerson, Mor Geva
- **Year**: 2023 (July, Dec v2)
- **Source**: arXiv:2307.12976
- **GitHub**: https://github.com/edenbiran/RippleEdits
- **Dataset**: RippleEdits benchmark (5K factual edits in `code/RippleEdits/data/benchmark/`)
- **Key Contribution**: Formal definition of ripple effects in knowledge editing. Proposes six evaluation criteria for ripple effects. Shows current KE methods fail to propagate updates consistently.
- **Methodology**:
  - Facts as triplets (entity, relation, object): (e, r, o) → (e, r, o*)
  - Knowledge graph K of factual triplets; ripple effect R(δ) = set of triplets that must be injected/modified/deleted
  - Six evaluation criteria:
    1. Logical Generalization (LG): symmetric/transitive relation constraints
    2. Compositionality I (CI): editing one edge → updating compositions with other edges
    3. Compositionality II (CII): cross-subject composition (other entity referencing edited entity)
    4. Subject Aliasing (SA): alias entities must also be updated
    5. Preservation (PV): one-to-many relations retain existing objects
    6. Relation Specificity (RS): unrelated relations unchanged
  - Benchmark: RECENT (newly-added facts), RANDOM, POPULAR subsets
  - Severity metric: number of facts affected by a single edit
- **Results**: All tested KE methods fail to consistently propagate ripple effects. In-context editing baseline outperforms parametric methods. Larger models handle ripple effects better.
- **Relevance**: **Direct methodological template** for cascading memory invalidation evaluation. The six ripple effect criteria (LG, CI, CII, SA, PV, RS) map to typed dependency edge types in the research hypothesis. Logical Generalization → LOCATED_IN structural edges; Compositionality → semantic CONFLICTS_WITH chains. The severity metric → invalidation cascade depth.

### Paper 6: MemoryBank — Enhancing Large Language Models with Long-Term Memory
- **Authors**: Wanjun Zhong, Lianghong Guo, Qiqi Gao, He Ye, Yanlin Wang
- **Year**: 2023 (May)
- **Source**: arXiv:2305.10250
- **GitHub**: https://github.com/zhongwanjun/MemoryBank-SiliconFriend
- **Key Contribution**: Novel memory mechanism with Ebbinghaus Forgetting Curve-inspired temporal decay. Three-pillar architecture: memory storage, memory retrieval, memory updater.
- **Methodology**:
  - Memory storage: timestamped conversations + hierarchical event summaries + dynamic user personality portraits
  - Ebbinghaus forgetting curve: memory strength decays exponentially with time; reinforced by repeated access
  - Memory retrieval: similarity-based recall; retrieved memories prepended to prompt
  - Evaluation: 15 virtual users, 10-day conversations, 194 probing questions
- **Results**: Demonstrated long-term memory recall on SiliconFriend chatbot. Works with both open-source (ChatGLM, BELLE) and closed-source (ChatGPT) models.
- **Relevance**: Provides the temporal decay mechanism for implicit preference drift (research hypothesis scenario 2). Ebbinghaus forgetting curve is a principled foundation for downweighting memories. However, MemoryBank lacks typed dependency edges and cascading propagation — it treats each memory independently without graph structure.

### Paper 7: PAMU — Preference-Aware Memory Update for Long-Term LLM Agents
- **Authors**: Haoran Sun, Zekun Zhang, Shaoning Zeng
- **Year**: 2025 (Oct)
- **Source**: arXiv:2510.09720
- **Key Contribution**: Sliding Window (SW) + Exponential Moving Average (EMA) for detecting preference drift in long-term dialogue. 5-dimensional preference vector (tone, length, emotion, density, formality).
- **Methodology**:
  - Preference extractor: RoBERTa for tone classification, emotion analysis, OpenIE for information density
  - SW + EMA fusion: wˆ_t = λ·SW_t + (1-λ)·EMA_t captures both short-term fluctuations and long-term trends
  - Change detection signal: deviation between SW and EMA estimates triggers memory update
  - Evaluated on LoCoMo dataset (5 task scenarios) against 5 baselines
- **Relevance**: Directly addresses the implicit preference drift problem (research hypothesis scenario 2). The SW+EMA mechanism is a quantitative approach to detecting when preference has shifted sufficiently to trigger cascading invalidation. Provides the behavioral co-occurrence signal alternative to LLM inference for establishing CONFLICTS_WITH edges.

### Paper 8: ChainEdit — Propagating Ripple Effects in LLM Knowledge Editing
- **Authors**: (from arXiv:2507.08427)
- **Year**: 2025 (July)
- **Source**: arXiv:2507.08427
- **Key Contribution**: Combines knowledge graph-derived logical rules with LLM reasoning to enable systematic chain updates for knowledge editing. Addresses the multi-hop propagation problem.
- **Relevance**: Provides a hybrid approach (structural rules + LLM reasoning) for establishing CONFLICTS_WITH edges — exactly the open question in the research hypothesis.

### Paper 9: PersistBench — When Should Long-Term Memories Be Forgotten by LLMs?
- **Authors**: (from arXiv:2602.01146)
- **Year**: 2026 (Feb)
- **Source**: arXiv:2602.01146
- **Key Contribution**: First benchmark measuring safety risks of memory persistence in conversational assistants. Identifies scenarios where persistence of outdated memories introduces harm.
- **Relevance**: Provides the safety framing for memory invalidation — outdated memories that should be invalidated but persist can cause harm. This motivates the cascading invalidation research.

## Common Methodologies

- **Typed dependency edges**: Used in HorizonBench (mental state graph) and conceptually in LoCoMo (temporal event graphs). Implementation in `src/causal_framework/evolution/relation_propagator.py` uses a weighted adjacency list with strength thresholds and decay factors.
- **Temporal decay for memory**: Ebbinghaus Forgetting Curve (MemoryBank), staleness thresholds (HorizonBench), EMA-based drift detection (PAMU)
- **LLM-as-oracle for semantic inference**: PersonaMem-v2 (RFT for implicit preference extraction), HorizonBench (LLM-generated event mentions without explicit preference restatement)
- **Knowledge graph propagation**: RippleEdits (factual triplet ripple effects), HorizonBench (typed dependency edge propagation), ChainEdit (logical rule + LLM reasoning chains)

---

## Standard Baselines

- **Full context (direct)**: Pass entire conversation history to LLM — baseline for all tasks; limited by context window
- **RAG**: Retrieve top-k relevant memories via embedding similarity — LoCoMo shows RAG offers best compromise for QA tasks
- **MemGPT**: Hierarchical memory with working context + archival storage; no explicit dependency edges
- **MemoryBank**: Ebbinghaus-inspired temporal decay + user portrait summarization
- **PAMU**: SW+EMA preference drift detection on LoCoMo benchmark
- **In-context editing**: Best performing baseline on RippleEdits benchmark for ripple effect propagation

---

## Evaluation Metrics

- **Accuracy (% correct)**: Primary metric for HorizonBench, PersonaMem-v2 (5-option MCQ), and LoCoMo QA
- **Pre-evolution distractor selection rate**: HorizonBench metric for belief-update failure; target: ≤25% (uniform baseline)
- **Evolved vs. static accuracy gap (Δevo)**: HorizonBench; all 25 models show negative gap; target: close to 0
- **Recall / F1**: LoCoMo QA task; used for measuring memory retrieval accuracy
- **Ripple effect coverage**: RippleEdits six criteria (LG, CI, CII, SA, PV, RS) — percentage of implied edits correctly propagated
- **Severity / cascade depth**: Number of memories affected by a single root-node change

## Gaps and Opportunities

### Gap 1: Semantic CONFLICTS_WITH edge establishment
HorizonBench uses structural dependency edges (dietary preferences → restaurant choices) but does not address how semantic conflict edges (preferring quiet → going to bars) are established. This is the core open problem. **Approaches**:
- (a) LLM inference at write time: use LLM to identify conflicting preferences and write CONFLICTS_WITH edges
- (b) Behavioral co-occurrence signals: detect when two memories are frequently mentioned together before/after but rarely simultaneously (PAMU's EMA signal as proxy)

### Gap 2: Propagation completeness guarantees
RippleEdits shows existing KE methods fail to propagate all ripple effects. The cascade depth (propagation_depth=2 in HorizonBench's RelationPropagator) limits completeness. A complete graph traversal approach needs to balance thoroughness with hallucination risk.

### Gap 3: Evaluation of cascading invalidation on real conversational data
HorizonBench uses synthetic data; LoCoMo uses real-world-inspired but generated data. No dataset provides ground-truth cascading invalidation traces on naturalistic user interactions.

### Gap 4: Incremental vs. batch invalidation
All existing systems propagate changes at write time. Real-time streaming environments need incremental update protocols that don't require full graph recomputation.

## Recommendations for Our Experiment

### Primary datasets
1. **HorizonBench** (`stellalisy/HorizonBench`): Primary evaluation dataset — provides ground-truth typed dependency edges, preference evolution provenance, and controlled diagnostic dimensions (evolved vs. static, explicit vs. implicit, long vs. short context). The `has_evolved` and `distractor_letter` fields enable direct measurement of belief-update failure.
2. **LoCoMo** (`Aman279/Locomo`): Secondary evaluation — provides real-world-inspired long-term conversations for QA-style memory evaluation. Use PAMU's LoCoMo experiment setup as baseline.
3. **RippleEdits** (`code/RippleEdits/data/benchmark/`): Evaluation of structural edge propagation correctness using the six criteria (LG, CI, CII, SA, PV, RS).

### Recommended baselines
1. **Full context (direct)**: Pass entire conversation to LLM — strong upper bound given long context
2. **RAG (embedding similarity)**: Retrieve top-k relevant memories — HorizonBench shows this fails at belief update because it retrieves pre-evolution stated preferences
3. **MemGPT/Letta**: Hierarchical memory without typed edges — ablation for structural vs. flat memory
4. **MemoryBank**: Temporal decay without graph propagation — ablation for decay vs. structural invalidation
5. **PAMU**: SW+EMA preference drift detection — baseline for implicit preference drift detection

### Recommended metrics
1. **Accuracy on evolved preferences** (HorizonBench): Primary metric
2. **Pre-evolution distractor selection rate**: Measures belief-update failure specifically
3. **Δevo (evolved − static accuracy gap)**: Measures improvement in handling preference evolution
4. **Ripple effect propagation recall** (RippleEdits criteria): Measures structural cascade completeness

### Methodological considerations
- HorizonBench's `relation_propagator.py` provides a ready implementation of typed dependency edge propagation with configurable depth (2), decay (0.7), and threshold (0.2) — adapt this as the core component
- For CONFLICTS_WITH edges: start with LLM-at-write-time inference (approach a) as it's tractable; compare with PAMU's EMA signal (approach b) as behavioral proxy
- Use HorizonBench's expression tracking (staleness threshold = 30 days) as the criterion for when a memory is eligible for cascading invalidation review
- Evaluate on both explicit and implicit preference variants (HorizonBench provides both via controlled expression rewrites) to quantify the semantic bridging problem
